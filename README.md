# fastest

> **Note:** This README was generated by an AI from the project headers and source.
> **Toy project** — built for personal learning, no stability guarantees and no intent for external use. That said, PRs are welcome.

A macro-based C testing framework built around a global test scheduler. Tests register themselves before `main()` via `__attribute__((constructor))` and can be run from C or Python. Nanosecond-precision timing and memory tracking included; zero external dependencies for the core library.

## Integration

### C-only binary

Copy the library and headers, then compile:

```bash
cp build/libfastest.a /your/project/lib/
cp -r include/fastest /your/project/include/

gcc -std=c11 -o tests tests.c -Iinclude -Llib -lfastest
```

Add `-DDEBUG` to enable `DEBUG_PRINTF` output.

### Python bindings (shared library)

The demo uses a shared Makefile that expects this layout inside your project directory:

```
your-project/
├── src/                   # your .c source files
├── vendor/
│   ├── bindings/          # symlink or copy of fastest/bindings/
│   │   └── pybind.cpp
│   ├── fastest/
│   │   └── libfastest.a   # symlink or copy of fastest/build/libfastest.a
│   └── include/
│       └── fastest/       # symlink or copy of fastest/include/fastest/
└── Makefile -> /path/to/fastest/demo/shared_Makefile
```

The symlink approach used in the demo keeps things tidy:

```bash
ln -s /path/to/fastest/demo/shared_Makefile Makefile
mkdir -p vendor/bindings vendor/fastest vendor/include
ln -s /path/to/fastest/bindings           vendor/bindings
ln -s /path/to/fastest/build/libfastest.a vendor/fastest/libfastest.a
ln -s /path/to/fastest/include/fastest    vendor/include/fastest
```

Then build:

```bash
make shared              # produces build/release/fastest.so
make BUILD=debug shared  # debug build with -g -O0 -DDEBUG
```

The Makefile compiles all `.c` files under `src/` as PIC objects, then links them with `vendor/bindings/pybind.cpp` and `libfastest.a` into the shared library. The output is named `fastest.so` so it can be imported directly:

```python
# from build/release/, or with fastest.so on PYTHONPATH
import fastest
```

## Testing Modes

fastest provides four ways to define and run tests, ranging from one-liners to fully self-contained inline blocks.

### 1. QUICKTEST — Expression-based

Tests a boolean expression directly. Timing is measured around the expression.

```c
FASTEST_QUICKTEST(name, expr, flags)
```

```c
FASTEST_QUICKTEST("add is 5", add(2, 3) == 5,
    FASTEST_ASSERT_EQ | FASTEST_FAIL_ERROR | FASTEST_TIME_NS);
```

The expression is evaluated; if it's truthy the test passes, otherwise `FASTEST_ERROR_ASSERT` is set. Result is printed immediately — this mode does **not** register in the scheduler.

---

### 2. CUSTOMTEST — Scheduler-registered function

Registers a test function in the global scheduler. The function receives a `FASTEST_TestOutput *out` pointer and sets flags and status on it manually.

```c
FASTEST_CUSTOMTEST(name, preload_flags, test_func, callback)
```

```c
void my_test(FASTEST_TestOutput *out) {
    out->test_flags |= FASTEST_ASSERT_EQ;
    out->exit_status |= add(2, 3) == 5 ? FASTEST_SUCCESS : FASTEST_ERROR_ASSERT;
    out->exit_status |= FASTEST_DEFAULT_LOG;
}

FASTEST_CUSTOMTEST("my test", FASTEST_TIME_NS | FASTEST_FAIL_ERROR, my_test, NULL);
```

`preload_flags` are merged into `out->test_flags` before the test runs. `callback` is an optional `void (*)(FASTEST_TestOutput *)` invoked after execution; pass `NULL` to skip.

The macro uses `__attribute__((constructor))`, so the test is registered before `main()`.

---

### 3. CUSTOMTEST\_INLINE — Inline body, external callback

Same as CUSTOMTEST but the test body is written inline in the macro, avoiding a separate named function.

```c
FASTEST_CUSTOMTEST_INLINE(name, preload_flags, callback, { body })
```

```c
void after_test(FASTEST_TestOutput *out) {
    DEBUG_PRINTF("%s done", out->test_name);
}

FASTEST_CUSTOMTEST_INLINE("inline add", FASTEST_TIME_NS | FASTEST_FAIL_ERROR,
    after_test,
    {
        out->test_flags |= FASTEST_ASSERT_EQ;
        out->exit_status |= add(2, 3) == 5 ? FASTEST_SUCCESS : FASTEST_ERROR_ASSERT;
        out->exit_status |= FASTEST_DEFAULT_LOG;
    }
)
```

---

### 4. CUSTOMTEST\_DINLINE — Fully inline (body + callback)

Both the test body and the callback body are written inline. Most compact form, no external functions needed.

```c
FASTEST_CUSTOMTEST_DINLINE(name, preload_flags, { callback_body }, { test_body })
```

```c
FASTEST_CUSTOMTEST_DINLINE("dinline add",
    FASTEST_TIME_NS | FASTEST_FAIL_ERROR | FASTEST_DEFAULT_LOG,
    {
        DEBUG_PRINTF("%s done, assert failed: %d",
            out->test_name,
            (out->exit_status & FASTEST_ERROR_ASSERT) > 0);
    },
    {
        out->test_flags |= FASTEST_ASSERT_EQ;
        out->exit_status |= add(2, 3) == 5 ? FASTEST_SUCCESS : FASTEST_ERROR_ASSERT;
        out->exit_status |= FASTEST_DEFAULT_LOG;
    }
)
```

### Running Scheduled Tests

Tests registered with CUSTOMTEST / INLINE / DINLINE are stored in the global scheduler (`FASTEST_list_t`). You can run them by name or index from anywhere, including `main()`:

```c
#include "fastest/test_list.h"

int main(void) {
    FASTEST_list_t *list;
    FASTEST_list_getInstance(&list);

    FASTEST_list_exec_name(list, "my test");   // run by name
    FASTEST_list_exec(list, 0);                // run by index
    return 0;
}
```

Tests run by the scheduler are timed automatically; you do not need to call `timespec_get` yourself.

## Flags Reference

Flags are `uint64_t` bitmasks combined with `|`. They live in `tests.h`.

### Assertion

| Flag | Assertion |
|------|-----------|
| `FASTEST_ASSERT_EQ` | `==` |
| `FASTEST_ASSERT_NEQ` | `!=` |
| `FASTEST_ASSERT_GT` | `>` |
| `FASTEST_ASSERT_GE` | `>=` |
| `FASTEST_ASSERT_LT` | `<` |
| `FASTEST_ASSERT_LE` | `<=` |

### Failure reporting

| Flag | Output style |
|------|-------------|
| `FASTEST_FAIL_ERROR` | Red `[FASTEST ERROR]` |
| `FASTEST_FAIL_WARNING` | Yellow `[FASTEST WARNING]` |
| `FASTEST_FAIL_LOG` | Blue `[FASTEST LOG]` |

### Timing

| Flag | Unit |
|------|------|
| `FASTEST_TIME_NS` | nanoseconds |
| `FASTEST_TIME_US` | microseconds |
| `FASTEST_TIME_MS` | milliseconds |
| `FASTEST_TIME_S` | seconds |

### Misc

| Flag | Effect |
|------|--------|
| `FASTEST_MEM_TRACK` | Track allocation/deallocation, report leaks |
| `FASTEST_DEFAULT_LOG` | Enable `FASTEST_Print_result` output |

### Status codes

Set on `out->exit_status` inside test functions.

| Code | Meaning |
|------|---------|
| `FASTEST_SUCCESS` | Test passed |
| `FASTEST_ERROR_ASSERT` | Assertion failed |
| `FASTEST_ERROR_MEMORY` | Memory leak detected |
| `FASTEST_ERROR_TIMEOUT` | Test exceeded time limit |
| `FASTEST_ERROR_EXCEPTION` | Signal / fatal error |
| `FASTEST_ERROR_UNEXPECTED` | Unexpected result |
| `FASTEST_ERROR_RESOURCE` | I/O or resource failure |
| `FASTEST_ERROR_INTERNAL` | Framework misuse |
| `FASTEST_ERROR_COLLISION` | Duplicate test name |
| `FASTEST_ERROR_NOT_FOUND` | Test name not in scheduler |

## Python Bindings

fastest can expose the test scheduler to Python via pybind11. Build the shared library from the `demo/showcase` directory (see [Building](#building)), then import it:

```python
import fastest
```

### API

```python
# Get all registered tests as a list of dicts
fastest.get_tests()

# Get a single test by name
fastest.get_test("my test")
# Returns: {"test_name": ..., "status": ..., "test_flags": ...,
#           "exit_status": ..., "allocation": ..., "deallocation": ..., "time_ns": ...}

# Get all tests whose names start with "suite/"
fastest.get_subtests("suite")

# Run a test by name (executes it and updates scheduler state)
fastest.run_test("my test")

# Test name constants are available under fastest.tests
fastest.tests.my_test   # equals "my test" (slashes and dashes normalized to underscores)
```

### Example

```python
import fastest

print(fastest.get_test("dinline add"))   # inspect before running
fastest.run_test("dinline add")
print(fastest.get_test("dinline add"))   # inspect after (status, time_ns, exit_status updated)
```

## Examples

### Minimal test file

```c
#include "fastest/quick_tests.h"
#include "fastest/tests.h"

int add(int a, int b) { return a + b; }

int main(void) {
    FASTEST_QUICKTEST("add 2+3==5", add(2, 3) == 5,
        FASTEST_ASSERT_EQ | FASTEST_FAIL_ERROR | FASTEST_TIME_NS);
    return 0;
}
```

### Mixed modes

```c
#include "fastest/quick_tests.h"
#include "fastest/custom_tests.h"
#include "fastest/tests.h"
#include "fastest/test_list.h"

int multiply(int a, int b) { return a * b; }

// Registered before main(), runs via scheduler
FASTEST_CUSTOMTEST_INLINE("multiply/basic",
    FASTEST_TIME_NS | FASTEST_FAIL_ERROR,
    NULL,
    {
        out->test_flags |= FASTEST_ASSERT_EQ;
        out->exit_status |= multiply(3, 4) == 12 ? FASTEST_SUCCESS : FASTEST_ERROR_ASSERT;
        out->exit_status |= FASTEST_DEFAULT_LOG;
    }
)

int main(void) {
    // Quick inline check — not in scheduler
    FASTEST_QUICKTEST("3*4 == 12", multiply(3, 4) == 12,
        FASTEST_ASSERT_EQ | FASTEST_FAIL_ERROR | FASTEST_TIME_US);

    // Run the scheduled test
    FASTEST_list_t *list;
    FASTEST_list_getInstance(&list);
    FASTEST_list_exec_name(list, "multiply/basic");

    return 0;
}
```

### Memory tracking

```c
#include "fastest/custom_tests.h"
#include "fastest/tests.h"
#include <stdlib.h>

void mem_test(FASTEST_TestOutput *out) {
    out->test_flags |= FASTEST_MEM_TRACK | FASTEST_FAIL_ERROR;

    void *ptr = malloc(64);
    out->allocation += 64;

    free(ptr);
    out->deallocation += 64;

    out->exit_status |= FASTEST_SUCCESS | FASTEST_DEFAULT_LOG;
}

FASTEST_CUSTOMTEST("mem/alloc-free", 0, mem_test, NULL);
```
